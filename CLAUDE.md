# 競馬予想アプリ — プロジェクトルール

## プロジェクト概要
個人開発の競馬予想Webアプリ。無料枠のみで完結する構成（Vercel Hobby / Supabase Free）。

## 技術スタック（決定済み）
- フロント/バックエンド: Next.js（App Router）
- ホスティング: Vercel（Hobbyプラン）
- DB: Supabase（Postgres, Free tier）
- データ取得方法: JRA公式サイトを低頻度スクレイピング（下記ルール必須）
- 未決定: データモデル、予想ロジックの初期形
  → これらは次のタスクとして設計エージェントに相談する

### データ取得ルール（必須・自己責任前提）
JRA-VAN DataLab.は月額有料かつWindows専用のJV-Link(ActiveX/COM)依存でVercelでは動かせないため不採用。
netkeiba等は利用規約でスクレイピングを明確に禁止しておりbot検知も強化されているため不採用。
JRA公式サイトの低頻度スクレイピングを採用するが、必ず以下を守ること:
1. 取得頻度は最小限（1日数回まで）に抑える
2. 取得結果は必ずSupabaseにキャッシュし、同じデータの再取得を避ける
3. User-Agentを明示する
4. 実装前にrobots.txt・利用規約を都度自分で再確認する

### 補足: 自動化について
ユーザーは「自動化のしやすさ」を重視しており、GASにも慣れている（GASなら早く作れる）。
Next.js+Vercel+Supabase構成を選んだ後もこの優先度は変わらないため、
定期的なデータ取得（レース情報・オッズ更新等）は Vercel Cron Jobs で自動実行する設計にすること。
もしNext.js構成での自動化・運用が想定より重くなる場合は、GASへの切り替えも選択肢として再検討してよい。

## ゴール
思いついた新機能（予想ロジック、データ取得、UI改善など）を、思いついた時にすぐ
「設計 → 実装 → 動作確認」まで通せる状態にする。完璧な設計より「まず動く」を優先する。

## 窓口（このプロジェクトで会話する際のデフォルト挙動）

このプロジェクトでの会話は、まず「窓口」として振る舞う。ユーザーから来た内容を以下のロジックで
振り分け、必要に応じて Task ツールでサブエージェント（`.claude/agents/`）に委譲する。

### 優先順位ロジック
1. **致命的バグ**（アプリが動かない、予想ロジックが落ちる、デプロイが失敗する等）
   → `test-debug-agent` に再現・原因切り分けを依頼 → 必要なら `jissou-agent` で修正
2. **作りかけの機能の続き**
   → 現在の設計方針を確認し、`jissou-agent` に実装を依頼。設計方針が未定/曖昧なら先に `sekkei-agent`
3. **新しい思いつき・磨き込みアイデア**
   → まず `sekkei-agent` に技術方針を軽く固めさせてから `jissou-agent` へ
4. **上記に当てはまらない雑多なメモ**（「そのうちやりたい」レベルのアイデア等）
   → 実装に着手せず、Obsidian Vault の `00-Inbox/` にノートとして書き出す（下記フォーマット参照）。
     ユーザーへの確認は不要、書き出したら一言報告するだけでよい。

判断に迷う場合はユーザーに一言確認してから進めてよい（趣味プロジェクトなので厳密さより気軽さ優先）。

## エージェント構成

| エージェント | 定義ファイル | モデル | 役割 |
|---|---|---|---|
| 窓口 | (このCLAUDE.mdに基づくメインスレッドの挙動) | Sonnet | 振り分けのみ、実装はしない |
| 設計 | `.claude/agents/sekkei-agent.md` | Opus | アーキテクチャ・技術選定・データ構造の方針決定 |
| 実装 | `.claude/agents/jissou-agent.md` | Sonnet | 設計方針に沿ったコード生成 |
| テスト/デバッグ | `.claude/agents/test-debug-agent.md` | Sonnet | 再現 → 原因特定 → 修正案 |
| バックログ整理 | `.claude/agents/backlog-agent.md` | Sonnet | Obsidian Vaultの棚卸し（月1回目安） |

呼び出しは Task ツールでエージェント名を指定するか、内容から自動判断して委譲してよい。

### モデル割り当ての方針
- 設計・アーキテクチャ判断 → Opus（精度重視、頻度は低いのでコスト影響は小さい）
- 実装・大量コード生成 → Sonnet
- デバッグの壁打ち・ちょっとした修正 → Sonnet

## Obsidian連携

- 接続方法: **方法B（ファイルシステム直接読み書き）**。MCP接続は使わず、Vaultフォルダに直接
  ファイルを読み書きする。タグ検索・バックリンク操作はできない前提。
- Vaultパス: `~/Documents/ObsidianVault`
- フォルダの役割:
  - `00-Inbox/` — 窓口が雑多なメモを一次的に書き出す場所
  - `01-Projects/keiba-yosou-app/` — このプロジェクトの設計メモ・仕様・実装ログ
  - `02-Backlog/` — バックログ整理エージェントが棚卸しした後のアイデア置き場
- ノート命名規則: `YYYY-MM-DD-タイトル.md`
- フロントマター規約（すべてのノート作成時に必須）:
  ```yaml
  ---
  title: 
  created: YYYY-MM-DD
  tags: [競馬予想アプリ, ...]
  ---
  ```

## 運用ルール

- **リポジトリはGitHub公開前提**。ただし機密情報（APIキー等）は `.env` に分離し、
  `.gitignore` で除外する。リポジトリに直接書かない。
- 公開前提のため、コミット前に個人情報・APIキー・課金情報が混入していないか軽く確認する。
- 完璧な設計より「まず動く」を優先し、リファクタは後回しでよい。
- 技術スタックは無料枠で完結すること（例: Vercel Hobby, Supabase Free, 外部データ取得も無料APIの範囲）。
